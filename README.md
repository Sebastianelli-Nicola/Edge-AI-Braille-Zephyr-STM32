#  Edge AI Braille Recognition on STM32 (Zephyr RTOS)
A real-time Edge AI firmware for Braille recognition on STM32 microcontrollers. Built with Zephyr RTOS, TensorFlow Lite Micro, and a custom CNN.

This repository contains the source code, the Machine Learning pipeline, and the testing tools for an **Edge Artificial Intelligence** system capable of recognizing Braille characters in real-time. 

The project was developed as a thesis work and demonstrates the implementation of a Convolutional Neural Network (CNN) on a highly memory-constrained microcontroller (128 KB SRAM, 512 KB Flash), abandoning the classic *bare-metal* approach in favor of an architecture based on **Zephyr RTOS**.

## 🧠 Neural Model Architecture (Micro-Braille Net)

The model was designed from scratch to operate with a microscopic RAM footprint, integrating advanced architectural solutions:

* **Separable Convolutions Backbone:** Replacement of standard convolutions with Depthwise Separable Convolutions and removal of MaxPooling layers in favor of Strides=(2,2), drastically reducing the number of parameters and the impact on SRAM.
* **Pixel Attention Module:** Implementation of a custom layer to make the network focus solely on the three-dimensional micro-shadows of the Braille dots, ignoring the noise generated by the paper texture.
* **Multi-Task Strategy (Two Heads):**
  * *Confidence Head (Sigmoid):* Binary output that determines whether a character is present in the patch or just noise/background.
  * *Classification Head (Softmax):* 65-class network (alphabet, numbers, punctuation) activated only in case of high confidence.

## Firmware Optimizations and Zephyr RTOS

Deployment on a Deep Embedded device required the adoption of rigorous software micro-optimizations in C++:

* **Preemptive RTOS Scheduling:** Management of concurrent tasks via Zephyr. The diagnostic task (Blinky LED) was configured with high priority (Priority 4) to prevent Starvation during heavy tensor processing blocks (CPU-bound).
* **Static Memory Management:** Allocation of the Tensor Arena surgically limited to 80 KB and memory-aligned (`alignas(16)`) to speed up processor fetch, manually instantiating only the necessary operators via `MicroMutableOpResolver`.
* **Zero-Copy In-Place Processing:** Serial acquisition of images directly populates the AI model's input buffers, eliminating RAM waste and avoiding heavy `memcpy()` calls.
* **Tight Spinloop Synchronization:** Data reception via UART at 115200 baud without Hardware Buffer Overrun, exploiting direct polling of physical registers without scheduler context-switches.
* **Hardware FPU Acceleration:** Replacement of software divisions with pre-calculated multiplications and use of the native 32-bit `expf()` function (Single Precision) for Sigmoid calculation, natively exploiting the M4F math coprocessor.

## Performance and Results (Stress Test)

The system was validated using a Python script (Hardware-in-the-Loop) that simulates a camera video stream, sending 1543 test patches ($32 \times 32$ pixels) directly to the board via UART.

| Metric | Result (STM32 Nucleo F446RE) |
| :--- | :--- |
| **CPU Frequency** | 180 MHz (No hardware NPU) |
| **RAM Usage (Tensor Arena)** | ~80 KB |
| **Average Inference Latency** | ~560 ms / image |
| **Global Accuracy on Test Set** | 80.17 % |
| **Recognized Classes** | 65 (Alphabet, Numbers, Braille Punctuation) |

## Future Developments

Overcoming current architectural limitations paves the way for the industrialization of a stand-alone assistive Wearable Device:

1. **DCMI and DMA:** Internalization of visual acquisition by interfacing a camera module (e.g., OV7670) via Direct Memory Access, reducing CPU load for image transfer to zero.
2. **Active Grazing Illumination:** Hardware integration of grazing-incidence micro-LEDs driven via GPIO to force the projection of sharp shadows on Braille dots, making the model immune to ambient light.
3. **Quantization Aware Training (QAT):** Network training simulating 8-bit precision loss directly in the training epochs, enabling the use of int8 models without suffering the performance degradation of classic quantization.

## How to reproduce the project

**1. Firmware Configuration:**
* Install the Zephyr Project ecosystem and the west meta-tool.
* Navigate to the `firmware_STM32/` directory and build:
  ```bash
  west build -p always -b nucleo_f446re
  ```

**2. Flashing the Firmware to the Board:**
* Once the build is complete, locate the generated binary file (`zephyr.bin` in the build folder).
* Connect the STM32 board to the PC via a USB cable.
* Perform the automatic firmware flash directly from the terminal by running the following command:
  ```bash
  py -m west flash
  ```
* Alternatively, drag and drop (or copy-paste) the file directly into the `NODE_F446RE` virtual disk drive created by the PC when the STM32 board is connected via USB.

**3. Running the Test:**

* Install the necessary Python dependencies for the Host environment:
  ```bash
  pip install -r host_tools_PC/requirements.txt
  ```

* Launch the simulation (make sure to enter the correct COM port inside the `invia_foto_carattere.py` file):
  ```bash
  python host_tools_PC/invia_foto_carattere.py
